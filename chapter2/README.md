## 2.1 리팩터링 정의

리팩터링이라는 용어는 **명사**와 **동사** 두 가지로 쓰인다.

### 명사로서의 리팩터링

> **리팩터링(명사):** 소프트웨어의 겉보기 동작은 그대로 유지한 채, 코드를 이해하고 수정하기 쉽도록 내부 구조를 변경하는 것

예: "함수 추출하기", "조건문 분해하기" 등 개별 리팩터링 기법을 가리킨다.

### 동사로서의 리팩터링

> **리팩터링하다(동사):** 소프트웨어의 겉보기 동작은 그대로 유지한 채, 여러 가지 리팩터링 기법을 적용해서 소프트웨어를 재구성하다

핵심은 **"겉보기 동작(observable behavior)을 유지한다"** 는 것이다. 리팩터링 전후로 코드가 똑같이 동작해야 한다. 사용자 관점에서 달라지는 것이 없어야 한다.

### 리팩터링 vs 재구성(restructuring)

- **재구성**: 코드 베이스를 정리하거나 구조를 바꾸는 모든 행위를 포괄하는 일반적 용어
- **리팩터링**: 재구성 중에서도 **특정한 방식**으로 수행하는 것. 즉 겉보기 동작을 유지하면서 **작은 단계들을 순서대로** 연결해 큰 변화를 만들어 내는 것

리팩터링 과정에서 코드가 깨지는 순간이 없어야 한다. 각 단계가 아무리 작더라도 **언제든 멈출 수 있고**, 중간 상태에서도 코드는 정상 동작해야 한다.

## 2.2 두 개의 모자

켄트 벡(Kent Beck)이 제안한 비유로, 소프트웨어 개발 시 **"기능 추가"** 모자와 **"리팩터링"** 모자 중 하나만 쓴다는 것이다.

| 기능 추가 모자 🎩 | 리팩터링 모자 🎩 |
|---|---|
| 기존 코드를 건드리지 않고 **새 기능만 추가** | 기능을 추가하지 않고 **코드 구조만 개선** |
| **테스트를 추가**하고 통과시키는 데 집중 | 기존 테스트가 **모두 통과하는 상태** 유지 |

- 한 번에 한 모자만 쓴다.
- 두 모자를 **자주 바꿔 쓴다.**
- 지금 자신이 어떤 모자를 쓰고 있는지 **명확히 인식**해야 한다.

기능을 추가하다가 코드 구조를 바꾸고 싶어지면, 하던 일을 멈추고 리팩터링 모자로 바꿔 쓴 뒤 구조를 개선한다. 그 다음 다시 기능 추가 모자로 돌아온다.

## 2.3 리팩터링하는 이유

### 리팩터링하면 소프트웨어 설계가 좋아진다

- 리팩터링하지 않으면 소프트웨어의 내부 설계(아키텍처)가 서서히 썩어간다.
- 코드만으로 설계를 파악하기 어려워지면 설계를 유지하기 더 어려워지고, 설계가 부패하는 속도는 가속된다.
- 규칙적인 리팩터링은 코드의 구조를 지탱해준다.
- 중복 코드를 제거하면 모든 코드가 고유한 일을 수행함을 보장할 수 있고, 이는 바람직한 설계의 핵심이다.

### 리팩터링하면 소프트웨어를 이해하기 쉬워진다

- 프로그래밍은 컴퓨터만을 위한 것이 아니다. **다른 프로그래머(미래의 나 포함)** 도 읽는다.
- 코드를 이해하기 쉽게 만드는 것은 **"다른 사람을 배려하는 것"** 이다.
- 리팩터링은 코드의 목적이 더 잘 드러나게 해준다.
- 코드를 보고 파악한 것을 곧바로 코드에 반영하면, 나중에 다시 분석할 필요가 없다.

### 리팩터링하면 버그를 쉽게 찾을 수 있다

- 코드를 이해하기 쉬워지면 버그도 쉽게 찾을 수 있다.
- 리팩터링을 통해 코드의 의도를 명확히 드러내면, 프로그램의 구조를 명확히 다듬을수록 **"이 코드가 이렇게 동작해야 하는데, 실제로는 저렇게 동작하고 있네"** 라는 사실을 발견하기 쉽다.
- 켄트 벡의 표현: *"나는 뛰어난 프로그래머가 아니라 좋은 습관을 가진 괜찮은 프로그래머일 뿐이다."*

### 리팩터링하면 프로그래밍 속도를 높일 수 있다

- 앞의 장점들을 종합하면 결국 **개발 속도를 높여준다.**
- 내부 설계가 잘 된 소프트웨어는 새 기능을 추가할 지점과 어떻게 고칠지를 쉽게 찾을 수 있다.
- 모듈화가 잘 되어 있으면 전체 코드 중 일부만 이해하면 되므로 수정이 쉽다.
- **설계 지구력 가설(Design Stamina Hypothesis)**: 내부 설계에 심혈을 기울이면 소프트웨어의 누적 기능량이 시간이 갈수록 더 빠르게 증가한다. 설계가 나쁘면 처음엔 빠르지만 금세 생산성이 떨어진다.

```
기능 누적량
  ^
  |          /  좋은 설계
  |        /
  |      / -------- 나쁜 설계
  |    / /
  |  //
  | /
  +-------------------> 시간
```

## 2.4 언제 리팩터링해야 할까?

### 3의 법칙 (Rule of Three)

돈 로버츠(Don Roberts)가 제안한 가이드라인:

1. 처음에는 그냥 한다.
2. 비슷한 일을 두 번째로 하게 되면, 일단 계속 진행한다 (중복이 생겨도).
3. 비슷한 일을 **세 번째** 하게 되면 리팩터링한다.

> *"삼진이면 리팩터링하라(Three strikes and you refactor)."*

### 준비를 위한 리팩터링: 기능을 쉽게 추가하게 만들기

- 기능을 추가하기 직전에, 구조를 살짝 바꾸면 다른 작업이 훨씬 쉬워지는 경우가 많다.
- 예: 현재 코드가 이 기능을 추가하기에 불편한 구조라면 먼저 리팩터링한다.
- 비유: *"캠핑장을 떠날 때 처음 왔을 때보다 깨끗하게"* (보이스카웃 규칙)

### 이해를 위한 리팩터링: 코드를 이해하기 쉽게 만들기

- 코드를 읽으면서 이해한 것을 코드에 반영한다.
- 변수 이름 변경, 긴 함수 쪼개기 등 간단한 리팩터링만으로도 이해도가 크게 향상된다.
- 코드를 분석해서 얻은 통찰을 머릿속이 아닌 **코드 자체에** 녹여낸다.

### 쓰레기 줍기 리팩터링

- 코드를 파악하던 중 비효율적이거나 지저분한 코드를 발견하면 간단히 정리한다.
- 쉽게 고칠 수 있으면 즉시 고치고, 시간이 걸릴 것 같으면 메모해두고 하던 일을 마친 뒤 처리한다.
- **캠핑 규칙**: 항상 처음 왔을 때보다 깔끔하게 정리하고 떠난다.

### 계획된 리팩터링과 수시로 하는 리팩터링

- 리팩터링은 별도의 시간을 내서 하는 것이 아니라, **프로그래밍 과정에 자연스럽게 녹아들어야** 한다.
- 보통 기능 추가나 버그 수정을 하면서 함께 진행한다.
- 계획된 리팩터링이 필요할 때도 있지만, 대부분은 **기회가 될 때마다 수시로** 하는 것이 좋다.
- 리팩터링 커밋을 기능 추가 커밋과 분리할 수도 있지만, 반드시 그래야 하는 것은 아니다.

### 오래 걸리는 리팩터링

- 대규모 리팩터링도 **팀 전체가 달려들기보다**, 해당 영역을 건드릴 때마다 조금씩 개선하는 전략이 효과적이다.
- 예: 라이브러리 교체 시, 기존 것과 새 것을 모두 포용하는 추상 인터페이스를 만들어 놓고 점진적으로 마이그레이션한다.

### 코드 리뷰에 리팩터링 활용하기

- 코드 리뷰 시 리팩터링을 하면 개선안을 구체적으로 실험해볼 수 있다.
- 코드 작성자와 나란히 앉아서 하는 **짝 프로그래밍(pair programming)** 방식이 특히 효과적이다.

### 관리자에게 뭐라고 말해야 할까?

- 관리자가 기술에 정통하다면 리팩터링의 가치를 이해할 것이다.
- 그렇지 않다면? **말하지 않는다.** 리팩터링은 별도 일정이 필요한 작업이 아니다. 프로 개발자로서 효과적인 소프트웨어를 빠르게 만드는 데 필요한 활동일 뿐이다.

### 리팩터링하지 말아야 할 때

- 지저분한 코드를 발견해도 **굳이 수정할 필요가 없다면** 리팩터링하지 않는다. (외부 API처럼 호출만 하는 코드)
- **처음부터 새로 작성하는 게 쉬울 때**도 리팩터링하지 않는다.
- 리팩터링할지 새로 작성할지의 판단은 경험이 필요하다.

## 2.5 리팩터링 시 고려할 문제

### 새 기능 개발 속도 저하

- 리팩터링의 궁극적인 목적은 **개발 속도를 높이는 것** — 더 적은 노력으로 더 많은 가치를 만들기 위함이다.
- 리팩터링은 그 자체가 목적이 아니다. 경제적 이유로 하는 것이다.
- 리팩터링을 함으로써 **다른 작업이 빨라질 때만** 의미가 있다.
- "멋진 코드", "깔끔한 엔지니어링"에 집착하지 말자. 리팩터링의 본질은 **경제성**이다.

### 코드 소유권

- 코드 소유권이 엄격하면 리팩터링이 어렵다.
- 예: 함수 이름을 변경하고 싶은데 호출하는 쪽 코드를 수정할 권한이 없는 경우
- 해결: 기존 함수는 새 함수를 호출하는 래퍼(wrapper)로 남겨둔다.
- 팀 내 코드 소유권은 **느슨하게** 운영하는 것이 좋다. 팀원 누구나 수정할 수 있되 담당자에게 알리는 방식이 바람직하다.

### 브랜치

- 기능 브랜치(feature branch)를 오래 유지하면 **머지 통합이 어려워진다.**
- 리팩터링은 코드 전반을 건드리므로 머지 충돌 가능성이 높아진다.
- 해결: **CI(지속적 통합)** — 모든 팀원이 최소 하루에 한 번은 메인라인에 통합한다.
- CI를 하면 각자의 리팩터링을 빠르게 공유할 수 있고, 머지 충돌 가능성을 줄일 수 있다.
- 기능 브랜치를 쓰더라도 **수명을 짧게** 유지해야 한다 (최대 2~3일).

### 테스팅

- 리팩터링의 핵심 전제: **겉보기 동작이 변하지 않아야 한다.**
- 이를 검증하려면 **견고한 테스트 스위트(test suite)** 가 필수적이다.
- 테스트가 있으면 실수를 해도 금방 잡을 수 있으므로 리팩터링의 위험이 크게 줄어든다.
- 자가 테스트 코드(self-testing code)는 리팩터링의 핵심 전제 조건이다.
- 테스트 없이도 리팩터링은 가능하지만, 검증된 안전한 리팩터링 기법(도구 지원)만 사용해야 한다.

### 레거시 코드

- 레거시 코드는 대체로 **복잡하고, 테스트도 없으며, 많은 사람이 작성한** 코드다.
- 리팩터링이 가장 필요하지만 가장 어려운 상황이다.
- 해결: 테스트를 먼저 보강한다. 테스트 보강이 어려우면 **서로 관련된 부분끼리 나눠서** 테스트 가능한 부분부터 공략한다.
- 레거시 코드 활용 전략에 대해서는 마이클 페더스의 「레거시 코드 활용 전략(Working Effectively with Legacy Code)」을 참고하자.

### 데이터베이스

- 데이터베이스 스키마 변경도 리팩터링 가능하다.
- **점진적 마이그레이션**: 스키마 변경을 여러 단계로 나누어 프로덕션에 순차 릴리스한다.
- 예: 필드 이름 변경 시
  1. 새 필드 추가
  2. 기존 필드의 데이터를 새 필드로 복사
  3. 기존 필드를 참조하는 코드를 새 필드로 전환
  4. 기존 필드 제거
- 각 단계를 별도로 커밋하면 문제가 생겨도 쉽게 되돌릴 수 있다.

## 2.6 리팩터링, 아키텍처, 애그니(YAGNI)

리팩터링은 소프트웨어 아키텍처에 대한 사고 방식을 근본적으로 바꿔준다.

### 사전 설계의 한계

- 전통적 방식: 코딩 전에 아키텍처를 완벽히 설계한다.
- 문제: 요구사항은 **코딩을 시작하고 나서야** 진정으로 이해되는 경우가 많다.
- 요구사항이 변하면 사전에 세운 아키텍처가 맞지 않게 된다.

### 리팩터링이 가져다주는 유연성

- 리팩터링이 있으면 앞으로의 변경에 미리 대비해 유연성을 과도하게 심어 놓을 필요가 없다.
- 현재 요구사항만 해결하는 소프트웨어를 만들고, 진짜 필요해지면 그때 리팩터링으로 아키텍처를 바꾸면 된다.

### YAGNI (You Aren't Gonna Need It)

> "당장 필요하지 않은 기능은 만들지 마라."

- **간결한 설계(simple design)**, **점진적 설계(incremental design)** 라고도 한다.
- YAGNI는 "설계를 하지 말라"는 뜻이 아니다. **아키텍처를 리팩터링으로 변경할 수 있으므로**, 당장 필요한 것에만 집중하라는 뜻이다.
- 결과적으로 설계가 단순해지고, 복잡도가 낮아지며, 변경이 용이해진다.
- YAGNI를 실천하려면 리팩터링 역량이 뒷받침되어야 한다.

## 2.7 리팩터링과 소프트웨어 개발 프로세스

리팩터링의 실천은 다른 소프트웨어 개발 프랙티스들과 밀접하게 연관된다.

### 자가 테스트 코드 (Self-Testing Code)

- 리팩터링을 제대로 하려면 **자가 테스트 코드**가 반드시 뒷받침되어야 한다.
- 테스트가 없으면 리팩터링 중 실수를 잡아내기 어렵다.

### 지속적 통합 (Continuous Integration)

- 각 팀원의 리팩터링이 빠르게 통합되어야 충돌을 최소화할 수 있다.
- CI와 리팩터링은 서로 시너지 효과를 낸다.

### 세 가지 핵심 프랙티스의 상호 작용

```
자가 테스트 코드  ←→  리팩터링  ←→  지속적 통합
```

이 세 가지가 어우러지면:
- 각 프랙티스의 효과가 극대화된다.
- **YAGNI** 설계 방식이 가능해진다.
- 지속적 배포(Continuous Delivery)의 토대가 된다.

### 익스트림 프로그래밍(XP)과의 관계

- 리팩터링은 XP(Extreme Programming)에서 처음 체계화되었다.
- XP의 핵심 프랙티스(테스트, CI, 리팩터링)가 서로 지탱해주는 구조다.
- 이 프랙티스들이 갖춰지면 애자일 방법론의 다른 요소도 효과적으로 적용할 수 있다.

## 2.8 리팩터링과 성능

> "리팩터링하면 프로그램이 느려지지 않을까?"

### 직관적 우려

- 리팩터링은 코드를 이해하기 쉽게 만드는 방향으로 재구성하므로, 성능이 나빠지는 방향일 수도 있다.
- 예: 함수 호출 증가, 간접 참조 증가

### 실제로는

마틴 파울러는 세 가지 성능 최적화 방법을 소개한다:

**1. 시간 예산 배분 (Time Budgeting)**
- 설계 시 각 컴포넌트에 자원 예산(시간, 메모리)을 미리 배정한다.
- 하드 리얼타임 시스템에 적합하다.
- 일반적인 소프트웨어에는 과한 방법이다.

**2. 끊임없는 관심 (Constant Attention)**
- 모든 프로그래머가 항상 성능을 의식하며 코딩한다.
- 직관적이지만 **실제로는 비효과적**이다.
- 성능 변화의 대부분은 코드의 극히 일부분에서 발생하므로, 나머지 90%에 쏟는 노력은 낭비다.

**3. 의도적 성능 최적화 (Deliberate Optimization) ✅**
- 먼저 깔끔하게 구현한다 (리팩터링 포함).
- 성능 문제가 생기면 **프로파일러**로 병목 지점을 찾는다.
- 병목 지점만 집중 최적화한다.
- 리팩터링된 깔끔한 코드는 **프로파일링이 쉽고**, 병목을 찾기도 쉬우며, 최적화 적용도 쉽다.

### 핵심 원칙

> *"먼저 튜닝하기 쉬운 프로그램을 만들고, 그 다음에 원하는 속도가 나오게 튜닝하라."*

대부분의 경우 리팩터링으로 인한 성능 저하는 미미하며, 설사 저하되더라도 깔끔한 코드 구조 덕분에 성능 튜닝이 훨씬 쉬워진다. 결과적으로 **더 빠른 소프트웨어**를 만들 수 있다.

## 2.9 리팩터링의 유래

### 기원

- 리팩터링의 개념은 오래전부터 존재했지만, **용어와 체계**가 갖춰진 것은 1990년대다.
- **Smalltalk 커뮤니티**, 특히 워드 커닝엄(Ward Cunningham)과 켄트 벡(Kent Beck)을 중심으로 발전했다.

### 학문적 기여

- **빌 옵다이크(Bill Opdyke)**: 리팩터링에 관한 최초의 박사 논문(1992)을 작성했다. 의미 보존 리팩터링(semantics-preserving refactoring)을 체계화했다.
- **존 브랜트(John Brant)와 돈 로버츠(Don Roberts)**: 최초의 자동 리팩터링 도구인 **리팩터링 브라우저(Refactoring Browser)** 를 Smalltalk용으로 개발했다.

### 대중화

- **랄프 존슨(Ralph Johnson)** 그룹(일리노이 대학교)이 리팩터링 연구의 중심지였다.
- 마틴 파울러가 1999년 「리팩터링」 초판을 출간하면서 대중에 널리 알려졌다.
- 이후 여러 언어(Java, C#, Ruby 등)에 특화된 리팩터링 서적들이 출간되었다.

## 2.10 자동화된 리팩터링

### IDE의 리팩터링 지원

- 오늘날 대부분의 IDE는 자동 리팩터링 기능을 제공한다.
- 예: IntelliJ IDEA, Eclipse, Visual Studio, VS Code 등
- **이름 변경(Rename)**, **메서드 추출(Extract Method)**, **변수 인라인(Inline Variable)** 등이 대표적이다.

### 자동 리팩터링의 동작 방식

두 가지 접근 방식이 있다:

**1. 텍스트 기반 (Find & Replace)**
- 단순한 텍스트 검색 및 치환
- 안전하지 않다 — 동명의 다른 심볼까지 바꿀 수 있다.

**2. 구문 트리(AST) 기반 ✅**
- 코드를 구문 분석하여 구문 트리를 만들고, 이를 기반으로 변환한다.
- 타입 정보와 스코프를 이해하므로 훨씬 안전하고 정확하다.
- 정적 타입 언어에서 특히 효과적이다.

### 주의점

- 자동 리팩터링이 완벽하지 않을 수 있으므로, **테스트를 반드시 실행**해야 한다.
- 도구가 제공하는 리팩터링의 범위는 제한적이므로, 수동 리팩터링 역량도 여전히 중요하다.

## 2.11 더 알고 싶다면

리팩터링에 대해 더 깊이 공부하고 싶다면 다음 자료들을 참고하자:

| 자료 | 저자 | 설명 |
|---|---|---|
| **Refactoring Workbook** | 빌 웨이크(Bill Wake) | 리팩터링 연습 문제집. 직접 실습하며 익힐 수 있다 |
| **Refactoring to Patterns** | 조슈아 케리에브스키(Joshua Kerievsky) | 리팩터링과 디자인 패턴의 연결 고리를 다룬다 |
| **Refactoring Databases** | 스콧 앰블러(Scott Ambler), 프라모드 사달게(Pramod Sadalage) | 데이터베이스 리팩터링을 체계적으로 다룬다 |
| **Refactoring HTML** | 엘리엇 러스티 해럴드(Elliotte Rusty Harold) | 웹 프론트엔드(HTML) 리팩터링 |
| **Working Effectively with Legacy Code** | 마이클 페더스(Michael Feathers) | 레거시 코드에 테스트를 넣고 리팩터링하는 전략 |

또한 **refactoring.com** 웹사이트에서 최신 리팩터링 기법 카탈로그와 관련 자료를 확인할 수 있다.
